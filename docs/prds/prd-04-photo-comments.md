# PRD-04：照片评论区

## 1. 背景与目标

目前照片详情页只有静态信息展示（拍摄日期、地点、上传者），缺少用户互动能力。参考抖音评论区的交互模式，在照片详情页底部增加评论功能，让用户可以围绕一张照片进行交流，增强社区互动感和用户粘性。

## 2. 入口

照片详情页（`pages/detail/detail`）底部，信息卡片下方。

- 固定在页面底部的评论输入栏（类似抖音底部的"说点什么..."输入条）
- 信息卡片下方展示评论列表区域，显示评论总数

## 3. 核心功能

### 3.1 评论列表

- 展示当前照片的所有评论，按时间倒序（最新在前）
- 每条评论包含：
  - 评论者头像
  - 评论者昵称
  - 评论内容（纯文本）
  - 评论时间（智能格式：刚刚 / x分钟前 / x小时前 / 昨天 / x月x日 / 2025-03-07）
  - 点赞数 + 点赞按钮
  - 回复数（有回复时显示"展开x条回复"）
- 分页加载：首次加载 20 条，滑到底部加载更多
- 评论区顶部显示"评论 (xx)"总数标题

### 3.2 发表评论

- 页面底部固定输入栏，点击后弹起键盘
- 输入框 placeholder："说点什么..."
- 右侧发送按钮，输入内容后高亮可点击
- 发送成功后：
  - 评论立即插入列表顶部（乐观更新，不等接口返回）
  - 评论总数 +1
  - 输入框清空，键盘收起
- 未登录用户点击输入框时，触发登录流程
- 评论内容限制：1~500 字，不能为空

### 3.3 回复评论（二级评论）

- 点击某条评论的"回复"按钮，输入框切换为回复模式
  - placeholder 变为"回复 @昵称"
  - 发送后作为该评论的子评论
- 子评论展示在父评论下方，缩进显示
  - 默认折叠，显示"展开x条回复"
  - 点击展开后显示所有子评论，可收起
  - 子评论格式："昵称 回复 @被回复者：内容"
- 子评论也支持回复（但只有两级，子评论的回复仍挂在同一个父评论下，通过 @提及 区分）

### 3.4 评论点赞

- 每条评论右侧有点赞按钮（爱心图标 + 数字）
- 点击切换点赞/取消点赞状态
- 点赞数实时更新（乐观更新）
- 已点赞状态：爱心变红/实心
- 未登录用户点击时触发登录

### 3.5 删除评论

- 长按自己的评论，弹出操作菜单"删除"
- 确认后删除，评论从列表移除，总数 -1
- 只能删除自己发的评论

### 3.6 空状态

- 没有评论时显示："还没有评论，来说两句吧"

## 4. 数据模型

### 4.1 评论表 `t_comment`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键，雪花ID |
| photo_id | BIGINT | 关联照片ID |
| user_id | BIGINT | 评论者ID |
| parent_id | BIGINT | 父评论ID（顶级评论为0） |
| reply_to_user_id | BIGINT | 被回复者ID（顶级评论为0） |
| content | VARCHAR(500) | 评论内容 |
| like_count | INT | 点赞数，默认0 |
| reply_count | INT | 回复数，默认0（仅顶级评论维护） |
| create_time | DATETIME | 创建时间，自动填充 |
| update_time | DATETIME | 更新时间，自动填充 |
| deleted | TINYINT | 逻辑删除，默认0 |

### 4.2 评论点赞表 `t_comment_like`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| comment_id | BIGINT | 评论ID |
| user_id | BIGINT | 点赞用户ID |
| create_time | DATETIME | 点赞时间 |

唯一索引：`(comment_id, user_id)`，防止重复点赞。

## 5. 接口设计

### 5.1 获取评论列表

```
GET /api/comment/list
```

参数：
- `photoId`：照片ID（必填）
- `page`：页码，默认1
- `size`：每页条数，默认20

返回：
```json
{
  "code": 200,
  "data": {
    "list": [
      {
        "id": "1001",
        "userId": "2001",
        "nickname": "用户A",
        "avatarUrl": "https://...",
        "content": "这个地方好美",
        "likeCount": 5,
        "liked": false,
        "replyCount": 3,
        "createTime": "2026-03-07T10:30:00",
        "replies": []
      }
    ],
    "total": 42,
    "hasMore": true
  }
}
```

说明：
- `liked` 字段表示当前用户是否已点赞（未登录时为 false）
- `replies` 首次返回空数组，通过单独接口加载

### 5.2 获取子评论（回复）

```
GET /api/comment/replies
```

参数：
- `commentId`：父评论ID（必填）
- `page`：页码，默认1
- `size`：每页条数，默认50

返回：
```json
{
  "code": 200,
  "data": {
    "list": [
      {
        "id": "1002",
        "userId": "2002",
        "nickname": "用户B",
        "avatarUrl": "https://...",
        "content": "确实很美",
        "likeCount": 1,
        "liked": false,
        "replyToNickname": "用户A",
        "createTime": "2026-03-07T11:00:00"
      }
    ],
    "hasMore": false
  }
}
```

### 5.3 发表评论

```
POST /api/comment/add
```

参数（JSON Body）：
- `photoId`：照片ID（必填）
- `content`：评论内容（必填，1~500字）
- `parentId`：父评论ID（回复时传，顶级评论不传或传0）
- `replyToUserId`：被回复者ID（回复时传）

需要登录（JWT Token）。

返回：新创建的评论对象。

### 5.4 删除评论

```
POST /api/comment/delete
```

参数：
- `commentId`：评论ID

需要登录，只能删除自己的评论。删除顶级评论时，其下所有子评论一并删除。

### 5.5 点赞/取消点赞

```
POST /api/comment/like
```

参数：
- `commentId`：评论ID

需要登录。已点赞则取消，未点赞则点赞（toggle 逻辑）。

返回：
```json
{
  "code": 200,
  "data": {
    "liked": true,
    "likeCount": 6
  }
}
```

## 6. 前端交互细节

| 场景 | 行为 |
|------|------|
| 进入详情页 | 加载照片信息 + 评论列表第一页 |
| 滑到评论区底部 | 自动加载下一页评论 |
| 点击底部输入栏 | 弹起键盘，聚焦输入框 |
| 发送评论 | 乐观插入列表顶部，失败时回滚并提示 |
| 点击"回复" | 输入框切换回复模式，placeholder 变为"回复 @昵称" |
| 点击键盘外区域 | 收起键盘，退出回复模式 |
| 点击"展开x条回复" | 请求子评论接口，展开显示 |
| 点击爱心 | 切换点赞状态，数字 ±1 |
| 长按自己的评论 | 弹出 wx.showActionSheet "删除" |
| 点击评论者头像 | 暂不处理（后续可跳转用户主页） |

## 8. 不做的事情（本期）

- 评论中 @提及 其他用户（仅回复时自动带 @）
- 评论中发图片/表情包
- 评论排序切换（热度/时间）
- 评论举报
- 评论置顶
- 消息通知（有人回复/点赞你的评论）

## 9. 技术要点

- 评论列表使用页面级滚动（和照片信息在同一个滚动容器内），不用单独的 scroll-view
- 底部输入栏使用 `position: fixed`，键盘弹起时需要处理 `adjust-position`
- 点赞使用乐观更新：先改 UI，再发请求，失败时回滚
- 评论 ID 使用雪花算法（和 Photo、User 一致）
- `t_comment` 的 `reply_count` 通过触发器或应用层维护（发表/删除子评论时 ±1）
- 接口权限：`/api/comment/list` 和 `/api/comment/replies` 公开（WebConfig 放行），其余需要登录
- 时间格式化在前端处理（和社区页复用同一套逻辑）

# PRD-06：时迹地图 Web 管理后台

## 1. 背景与目标

当前所有管理功能（举报审核、申诉处理、用户处罚、操作日志）都嵌在微信小程序内，存在以下问题：

- 小程序屏幕小，审核效率低，无法同时查看多维度信息
- 无法展示数据图表和运营看板
- 批量操作体验差，不支持键盘快捷操作
- 管理员和普通用户共用同一个入口，权限边界模糊
- 无法支撑多人协作审核、工单分配等运营场景
- 缺少用户管理、内容管理等平台级运营能力

本期目标：建设一个独立的 Web 管理后台，将所有平台运营能力从小程序迁移到 Web 端，形成"小程序负责用户体验，Web 后台负责平台运营"的标准架构。

## 2. 技术方案

### 2.1 前端技术栈

- React 18 + TypeScript
- Ant Design 5（企业级 UI 组件库）
- Vite（构建工具）
- React Router 6（路由）
- Axios（HTTP 请求）
- TanStack Query（React Query）— 服务端状态管理（请求缓存、自动刷新、乐观更新）
- Zustand — 客户端状态管理（用户信息、权限、主题、全局通知）
- ECharts / Ant Design Charts（数据可视化）

### 2.2 后端方案

复用现有 Spring Boot 后端，不新建服务。具体做法：

- 新增 `/api/admin/auth/login` 接口，支持管理员账号密码登录（独立于微信登录）
- 新增 `AdminWebAuthInterceptor`，校验 Web 端 JWT Token
- 现有 `/api/admin/**` 接口全部复用，仅需调整鉴权方式
- 新增少量接口（数据看板、用户管理、内容管理）
- 所有接口统一使用现有 `Result<T>` 包装格式（`{code, message, data}`），前端 Axios 拦截器统一解包

### 2.3 CORS 跨域配置

Web 管理后台部署在独立域名，所有请求触发跨域，必须配置：

- 后端新增 `CorsConfig` 配置类，允许管理后台域名（如 `https://admin.xxx.com`）
- 允许 Headers：`Authorization`, `Content-Type`, `X-Client-Type`
- 允许 Methods：`GET`, `POST`, `PUT`, `DELETE`, `OPTIONS`
- 不使用 credentials（Token 通过 Authorization Header 传递，不走 Cookie）
- 预检请求缓存：`Access-Control-Max-Age: 3600`

### 2.4 双认证体系共存

当前后端只有微信登录的 JWT 体系（`JwtInterceptor` / `OptionalJwtInterceptor`）。Web 端引入独立认证后，两套体系需要共存：

**区分方案：** 通过 HTTP Header `X-Client-Type` 区分来源。

- `X-Client-Type: web` → 使用 `AdminWebAuthInterceptor`，校验 Web JWT Token，解析出 `adminAccountId` 和 `role`
- 无此 Header 或值为 `miniprogram` → 使用现有 `JwtInterceptor`，校验微信 JWT Token，解析出 `userId`

**路径规则：**
- `/api/admin/**` 路径同时接受两种 Token（小程序管理页面仍用微信 JWT，Web 端用管理员 JWT）
- 拦截器根据 `X-Client-Type` 决定走哪条校验链路
- 两种 Token 使用不同的 JWT Secret 或不同的 `issuer` 字段，避免混淆

**兼容性保证：** 现有小程序管理页面调用 `/api/admin/**` 时不传 `X-Client-Type`，走原有微信 JWT 校验，完全不受影响。

### 2.5 部署方案

- Web 前端打包为静态文件，通过 Nginx 托管或放入 Spring Boot 的 static 目录
- 访问地址：`https://admin.你的域名.com` 或 `https://你的域名.com/admin/`

- 后端新增 `t_admin_account` 表存储管理员账号

## 4. 功能模块

### 4.1 登录页

- 账号 + 密码登录
- 登录失败 5 次锁定 15 分钟
- 登录成功后跳转数据看板
- 首次登录（使用初始密码）强制修改密码
- 密码复杂度要求：最少 8 位，必须包含大小写字母和数字
- 密码过期策略：90 天未修改密码时登录后强制修改
- 禁止使用最近 3 次历史密码
- 登录成功/失败均记录日志（含 IP、User-Agent）

#### 4.4.2 申诉详情页

- 原始被举报内容预览
- 原处理结果
- 操作区：采纳（恢复内容/撤销处罚）/ 驳回（填写原因）

**内容恢复机制：**

当前实现中，照片被删时调用了 `photoMapper.deleteById()`（逻辑删除）+ `cosService.deleteByUrl()`（COS 物理删除）。COS 文件一旦删除无法恢复，导致申诉采纳后无法恢复照片。

解决方案：
- 删除照片时不立即清理 COS 文件，仅做逻辑删除（`deleted=1`）
- 新增 COS 文件延迟清理机制：逻辑删除 30 天后，定时任务批量清理 COS 文件
- 申诉采纳后，将逻辑删除的记录恢复（`deleted=0`），照片重新可见
- 超过 30 天的申诉，因 COS 文件已清理，无法恢复照片，申诉页面展示"内容已过保留期，无法恢复"

**Tab 页签：**

- 照片网格，支持预览大图

Tab 2 — 用户评论：
- 评论列表，显示评论内容、所属照片、时间
- 可直接删除违规评论

Tab 3 — 违规记录：

- 每条记录：违规类型、原因、处罚措施、处理人、时间

Tab 4 — 举报记录：
- 该用户被举报的所有记录
- 该用户发起的所有举报

**操作区：**
- 处罚按钮：警告 / 禁言 / 禁止上传 / 封号
- 解除处罚按钮（如果当前有生效中的处罚）

**解除处罚实现：**
- 解除禁言：清空 `muteUntil`
- 解除禁止上传：清空 `banUploadUntil`
- 解除封号：`isBanned` 置 0
- 所有解除操作必须记录操作日志
- 解除后给用户发送通知："你的 XX 处罚已被解除"

**处罚生效检查（业务层拦截）：**
- 发表评论时：检查 `muteUntil` 是否在当前时间之后，是则拒绝并提示"你已被禁言，剩余 X 天"
- 上传照片时：检查 `banUploadUntil` 是否在当前时间之后，是则拒绝并提示
- 发送私信时：检查 `muteUntil`
- 所有操作时：检查 `isBanned`，是则拒绝并提示"账号已被封禁"
- 处罚到期自动失效（实时判断时间戳，无需定时任务清理）

## 8. 数据库新增

## 12. 安全要求

### 12.1 认证与授权
- 所有管理接口必须校验 Web JWT Token + 角色权限
- 密码使用 BCrypt 加密存储
- 登录失败 5 次锁定 15 分钟
- 密码复杂度：最少 8 位，包含大小写字母和数字
- 密码过期：90 天强制修改
- 首次登录强制修改初始密码
- 禁止使用最近 3 次历史密码

### 12.2 Token 存储方案
- 采用 localStorage + Authorization Header 方案（与现有小程序 JWT 方案一致）
- 不使用 httpOnly Cookie，因此天然免疫 CSRF 攻击，无需额外 CSRF Token 机制
- Token 格式：`Authorization: Bearer <web-jwt-token>`
- Web JWT 使用独立的 Secret，与小程序 JWT Secret 不同

### 12.3 操作审计
- 敏感操作（封号、删除内容、解除处罚）记录操作日志
- 登录成功/失败记录日志（含 IP、User-Agent、时间）
- 登出记录日志
- 密码修改记录日志
- 审计日志不可由当事人删除，仅 super_admin 可查看全量日志

### 12.4 权限粒度
- 现有 `AdminAuthService.requireAdmin()` 升级为 `requireRole(String... allowedRoles)`
- 每个 Controller 方法通过自定义注解 `@RequireRole("super_admin", "moderator")` 声明所需角色
- 拦截器统一校验角色权限

### 12.5 部署隔离
- 管理后台部署在独立域名或路径，与用户端隔离
- 管理后台域名不对外公开，仅内部人员知晓

### 12.6 管理员账号与小程序用户的关联
- `t_admin_account.linked_user_id` 用于关联小程序用户（可选）
- Web 端操作统一使用 `admin_account.id` 作为操作人标识
- 通知中展示的"处理人"使用 `admin_account.nickname`
- 关联后，管理员可在小程序端通过微信 JWT 访问管理页面，也可在 Web 端通过账号密码访问

## 13. 数据导出

运营场景中导出是刚需，以下列表页支持导出 Excel/CSV：

- 导出操作记录操作日志

## 17. Dashboard 查询性能

数据看板需要聚合查询多张表的 30 天数据，随数据量增长会成为性能瓶颈。

解决方案：
- 使用 Spring Cache 内存缓存，缓存有效期 10 分钟
- 看板页面显示"数据更新于 XX 分钟前"提示
- 后续数据量大时，可增加 `t_daily_stats` 统计快照表，每天凌晨定时任务预计算

## 18. 举报优先级与 SLA

- 同一内容被 3 人以上举报时，自动升级为高优先级
- 举报原因为"违法违规"或"色情低俗"时，自动标记为高优先级
- 待处理超过 24 小时的举报，列表中红色高亮显示
- 待处理超过 48 小时的举报，Dashboard 单独展示"超时未处理"计数

## 19. 举报原因可配置化

当前举报原因硬编码在小程序前端。需要实现完整的动态配置链路：

- 后端新增 `t_report_reason` 配置表（或使用 `t_system_config` 通用配置表）
- 新增 `/api/config/report-reasons` 公开接口，返回当前生效的举报原因列表
- 小程序启动时从后端拉取举报原因列表，缓存到本地
- Web 管理后台"系统配置"页面可增删改举报原因
- 修改后，小程序下次启动自动生效

# PRD-05.1：举报治理闭环（二期）

## 1. 背景与目标

一期已经解决“用户可举报、平台可审核、内容可处置、结果可反馈”的最小闭环问题，但随着内容规模和用户量上升，平台会面临更复杂的治理挑战：

- 举报量上升后，人工处理效率不足
- 单次只处理“内容”，缺少对违规用户的持续治理
- 误判后缺少申诉和复核机制
- 缺少审计记录和运营分析，无法评估治理质量
- 管理员权限过粗，存在操作风险

二期目标是将一期的最小闭环升级为“可规模化运营的内容治理系统”，建立更完整的审核、处罚、申诉、审计和数据分析能力。

## 2. 二期目标

- 建立更完善的管理员角色与权限体系
- 增加举报聚合、批量处理、优先级机制
- 增加对违规用户的处罚能力
- 增加被举报人通知与申诉入口
- 增加审核日志与操作留痕
- 增加治理数据看板与运营指标
- 增加敏感内容的自动识别与辅助审核能力

## 3. 二期定位

一期是“能上线的最小闭环”，二期是“能长期运营的治理平台”。

二期不推翻一期，而是在一期基础上扩展：

- 从单条举报处理，升级为案件化处理
- 从内容级治理，升级为内容 + 用户级治理
- 从人工单点操作，升级为流程化运营

## 4. 角色定义

### 4.1 普通用户

- 举报违规内容
- 查看举报处理结果
- 对被驳回或被处罚结果发起申诉（按权限）

### 4.2 举报人

- 除普通用户能力外，可查看自己举报的最终结果
- 可查看是否采纳、是否已处理

### 4.3 被举报人

- 可收到内容被处理通知
- 可查看被处理原因
- 可对处罚或内容下架发起申诉

### 4.4 审核员

- 查看举报案件
- 处理普通举报
- 执行标准处置动作

### 4.5 审核主管

- 复核争议案件
- 审批严重处罚
- 处理申诉

### 4.6 超级管理员

- 配置治理规则
- 分配角色权限
- 查看全量数据看板

## 5. 核心能力

### 5.1 举报聚合

同一对象在短时间内被多人举报时，不再只按单条记录独立处理，而是聚合为一个待处理案件。

聚合维度：

- `targetType`
- `targetId`
- 待处理状态

案件页展示：

- 举报总人数
- 举报原因分布
- 最近举报时间
- 高频举报描述

价值：

- 降低审核员重复劳动
- 快速识别高风险内容

### 5.2 优先级与队列

为举报案件增加优先级：

- P0：违法违规、涉政、色情未成年、极端风险
- P1：色情低俗、侵权、人身攻击
- P2：虚假信息、其他

队列规则：

- 默认按优先级 + 举报次数 + 举报时间排序
- 高优先级案件优先进入审核列表

### 5.3 用户处罚能力

在“处置内容”之外，增加对违规用户的治理动作：

- 警告
- 限制发布 1 天 / 3 天 / 7 天
- 禁言私信
- 封禁账号

适用场景：

- 多次被举报且违规成立
- 发布严重违规内容
- 恶意骚扰、辱骂、人身攻击

### 5.4 被举报人通知

当内容被平台处理时，被举报人也需要收到通知。

通知内容包括：

- 哪条内容被处理
- 处理原因
- 当前处罚动作
- 是否支持申诉

### 5.5 申诉流程

若用户认为处罚不合理，可发起申诉。

申诉流程：

```
被举报人收到处理通知
    │
    ▼
进入申诉页，填写说明
    │
    ▼
平台复核
    │
    ├── 维持原处理
    └── 撤销处理 / 恢复内容 / 解除处罚
```

申诉要求：

- 只允许对“已处理”案件发起申诉
- 每个案件仅允许发起一次申诉
- 申诉必须填写说明

### 5.6 批量处理

审核员支持对多条举报案件执行批量操作：

- 批量驳回
- 批量下架
- 批量标记重复举报
- 批量转交主管

### 5.7 审核日志与留痕

所有关键治理动作必须写入审计日志：

- 谁处理了哪条举报
- 在什么时间处理
- 做了什么动作
- 原因是什么
- 处理前后状态是什么

这部分用于：

- 风险追责
- 误操作排查
- 审核质量评估

### 5.8 数据看板

管理后台增加治理数据看板。

核心指标：

- 今日举报量
- 待处理举报量
- 平均处理时长
- 采纳率
- 驳回率
- 高风险原因分布
- 高频违规用户
- 高频被举报内容类型

## 6. 用户流程

### 6.1 举报聚合流程

```
多个用户举报同一内容
    │
    ▼
系统聚合为同一案件
    │
    ▼
案件进入待审核队列
    │
    ▼
审核员一次处理整组举报
```

### 6.2 治理处罚流程

```
审核员判定违规成立
    │
    ▼
下架内容 / 删除评论
    │
    ▼
判断是否需要处罚作者
    │
    ├── 是 → 警告 / 禁言 / 限制发布 / 封禁
    └── 否 → 仅处置内容
    │
    ▼
通知举报人 + 被举报人
```

### 6.3 申诉复核流程

```
被举报人发起申诉
    │
    ▼
主管进入申诉列表
    │
    ▼
复核内容与证据
    │
    ├── 维持原判
    └── 改判并恢复内容/撤销处罚
```

## 7. 数据模型扩展

### 7.1 举报表 `t_report`

在一期基础上增加：

| 字段 | 类型 | 说明 |
|------|------|------|
| case_id | BIGINT | 归属案件 ID |
| priority | TINYINT | 优先级 |
| source | VARCHAR(20) | 举报来源，如 photo_detail/comment_list |

### 7.2 举报案件表 `t_report_case`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| target_type | VARCHAR(20) | 举报对象类型 |
| target_id | BIGINT | 举报对象 ID |
| report_count | INT | 举报次数 |
| reporter_count | INT | 举报人数 |
| status | TINYINT | 待处理/已处理/已关闭/申诉中 |
| priority | TINYINT | 优先级 |
| final_action | VARCHAR(50) | 最终处理动作 |
| final_result | VARCHAR(500) | 最终处理说明 |
| assignee_id | BIGINT | 当前处理人 |
| handled_by | BIGINT | 最终处理人 |
| handled_time | DATETIME | 最终处理时间 |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

### 7.3 用户处罚表 `t_user_penalty`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| user_id | BIGINT | 被处罚用户 |
| case_id | BIGINT | 关联案件 |
| penalty_type | VARCHAR(30) | warning/mute/restrict_post/ban |
| start_time | DATETIME | 生效时间 |
| end_time | DATETIME | 结束时间，永久处罚可为空 |
| status | TINYINT | 生效中/已结束/已撤销 |
| reason | VARCHAR(500) | 处罚原因 |
| created_by | BIGINT | 创建人 |
| create_time | DATETIME | 创建时间 |

### 7.4 申诉表 `t_appeal`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| case_id | BIGINT | 关联案件 |
| user_id | BIGINT | 申诉人 |
| content | VARCHAR(1000) | 申诉说明 |
| status | TINYINT | 待复核/已维持/已改判 |
| result | VARCHAR(500) | 复核结果 |
| handled_by | BIGINT | 复核人 |
| handled_time | DATETIME | 复核时间 |
| create_time | DATETIME | 创建时间 |

### 7.5 审核日志表 `t_moderation_log`

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| case_id | BIGINT | 关联案件 |
| operator_id | BIGINT | 操作人 |
| action | VARCHAR(50) | 操作类型 |
| before_status | VARCHAR(50) | 操作前状态 |
| after_status | VARCHAR(50) | 操作后状态 |
| detail | VARCHAR(1000) | 操作详情 |
| create_time | DATETIME | 操作时间 |

## 8. 权限设计

二期升级为角色权限体系。

角色建议：

- `moderator`：审核员
- `moderator_lead`：审核主管
- `super_admin`：超级管理员

能力划分：

- 审核员：查看、处理普通举报
- 主管：复核申诉、执行严重处罚
- 超管：角色配置、规则配置、全量数据

## 9. 接口设计

### 9.1 案件列表

```
GET /api/admin/report-case/list
```

支持：

- 状态筛选
- 优先级筛选
- 对象类型筛选
- 指派人筛选

### 9.2 案件详情

```
GET /api/admin/report-case/detail/{id}
```

返回：

- 案件基础信息
- 聚合后的举报记录
- 内容详情
- 举报人列表
- 处罚历史
- 审核日志

### 9.3 指派案件

```
POST /api/admin/report-case/assign
```

### 9.4 批量处理

```
POST /api/admin/report-case/batch
```

### 9.5 用户处罚

```
POST /api/admin/penalty/create
```

### 9.6 发起申诉

```
POST /api/appeal/submit
```

### 9.7 获取我的申诉

```
GET /api/appeal/my
```

### 9.8 处理申诉

```
POST /api/admin/appeal/resolve
```

## 10. 管理后台页面

二期建议建设完整管理后台，至少包含：

- 举报案件列表页
- 举报案件详情页
- 申诉列表页
- 申诉详情页
- 用户处罚页
- 审核日志页
- 数据看板页

## 11. 自动化治理能力

二期引入“辅助审核”，不直接完全自动处罚。

可引入能力：

- 敏感词检测
- 文本辱骂识别
- 图片违规识别（接第三方审核服务）
- 高风险用户命中提醒
- 重复举报快速聚合

策略：

- 高风险内容自动打标
- 自动进入高优先级队列
- 默认仍由人工最终裁决

## 12. 交互细节

| 场景 | 行为 |
|------|------|
| 多人举报同一内容 | 后台聚合展示为同一案件 |
| 案件超时未处理 | 高亮提醒或进入超时列表 |
| 严重处罚动作 | 二次确认 |
| 用户被处罚 | 收到站内通知 |
| 用户申诉成功 | 恢复内容或撤销处罚，并通知用户 |
| 审核员误操作 | 可通过审核日志追踪 |

## 13. 指标与考核

建议统计以下治理指标：

- 举报处理 SLA
- 平均处理时长
- 举报采纳率
- 申诉率
- 申诉改判率
- 审核员处理量
- 审核员误判率
- 高风险内容命中率

## 14. 不做的事情（本期）

- 完全自动封禁
- 对外公开处罚公示
- 多级组织架构审批流
- 多租户治理系统

## 15. 技术要点

- 所有治理动作必须写审计日志
- 用户处罚需在内容发布、评论、私信等入口统一校验
- 申诉和案件处理需严格事务化
- 对象恢复能力要谨慎，若涉及物理删除则需提前设计恢复策略
- 高风险识别服务建议做成可插拔，不与主流程强耦合

## 16. 验收标准

- [ ] 同一内容的多次举报可聚合为案件
- [ ] 审核员可按优先级处理案件
- [ ] 平台可对违规用户执行处罚
- [ ] 被举报人可收到处理通知
- [ ] 被举报人可发起申诉
- [ ] 主管可处理申诉并维持或改判
- [ ] 所有关键治理动作都有审计日志
- [ ] 管理后台可查看核心治理指标
- [ ] 举报治理从“内容审核”升级为“内容 + 用户 + 申诉 + 审计”的完整运营体系

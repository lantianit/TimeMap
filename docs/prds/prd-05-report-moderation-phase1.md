# PRD-05：举报治理闭环（一期）

## 1. 背景与目标

地图时迹已经具备图片上传、评论、私信、通知等 UGC 能力，用户可以生产内容，也可以和他人的内容互动。当前系统虽然已有举报入口，但只有“提交举报”能力，平台侧缺少查看、审核、处理、反馈的完整流程，无法形成可运营、可追责、可持续的内容治理闭环。

一期目标是建设一个“最小可上线”的举报治理系统，确保从用户发起举报到平台处理完成，再到举报结果可查询，整条链路可跑通。

## 2. 一期目标

- 支持用户举报照片、评论
- 支持平台查看待处理举报列表
- 支持平台审核举报并执行处理动作
- 支持平台驳回无效举报
- 支持对违规照片、评论执行下架/删除处理
- 支持记录处理人、处理时间、处理结果
- 支持举报人查看自己的举报记录和处理状态
- 支持在站内通知举报人处理结果

说明：只有当“用户可提交 + 平台可处理 + 内容可处置 + 结果可回查”四段链路都打通时，才算举报流程闭环。

## 3. 角色定义

### 3.1 普通用户

- 可以对照片发起举报
- 可以对评论发起举报
- 可以查看自己提交过的举报记录
- 可以查看举报是否已处理、处理结果是什么

### 3.2 管理员

- 可以查看全部举报列表
- 可以按状态筛选待处理/已处理/已驳回举报
- 可以查看举报详情
- 可以执行“采纳并处理”或“驳回”操作
- 可以填写处理备注

### 3.3 系统

- 记录举报状态流转
- 记录处理人、处理时间、处理结果
- 在处理完成后给举报人发送站内通知

## 4. 核心问题

当前系统的主要问题：

- 用户举报后，平台不知道去哪里看
- 举报没有审核状态流转
- 举报被处理后，用户不知道结果
- 平台无法直接处置被举报内容
- 没有处理备注和责任归属，后续难以追踪

## 5. 核心功能

### 5.1 用户发起举报

- 举报对象支持两类：
  - 照片 `photo`
  - 评论 `comment`
- 用户在照片详情页、评论区对内容发起举报
- 举报原因使用预设选项：
  - 色情低俗
  - 违法违规
  - 侵权
  - 虚假信息
  - 人身攻击
  - 其他
- 允许填写补充说明（可选，最多 500 字）

规则：

- 必须登录后才能举报
- 不允许举报自己发布的内容
- 同一用户对同一对象重复举报时：
  - 若存在未处理举报，则提示“已举报，请等待处理”
  - 若已处理/已驳回，允许再次举报

### 5.2 我的举报记录

用户在个人中心新增“我的举报”入口，进入后可查看自己提交过的举报记录。

每条举报展示：

- 举报对象类型（照片/评论）
- 举报原因
- 举报时间
- 当前状态
- 处理结果摘要

状态文案：

- 待处理
- 已处理
- 已驳回

### 5.3 管理端举报列表

管理员可查看举报列表，默认展示“待处理”举报。

列表字段：

- 举报 ID
- 举报对象类型
- 举报对象 ID
- 举报原因
- 举报人
- 举报时间
- 当前状态

支持筛选：

- 按状态筛选：待处理 / 已处理 / 已驳回
- 按对象类型筛选：照片 / 评论

支持分页加载。

### 5.4 管理端举报详情

管理员点击举报列表项后进入详情页，查看完整信息。

详情内容包括：

- 举报基础信息
- 举报原因
- 补充说明
- 举报人信息
- 被举报对象信息
- 被举报对象发布者信息
- 当前状态
- 处理记录

对象详情要求：

- 若举报对象为照片，展示图片、地点、拍摄日期、上传者
- 若举报对象为评论，展示评论内容、评论者、所属照片 ID

### 5.5 采纳举报并处理内容

管理员可执行“采纳举报”。

采纳举报时需选择处理动作：

- 对照片：
  - 下架照片
- 对评论：
  - 删除评论

并填写：

- 处理备注（必填，便于后续回查）

处理完成后：

- 举报状态变为“已处理”
- 记录处理人、处理时间、处理备注
- 执行内容处置动作
- 通知举报人“举报已处理”

### 5.6 驳回举报

管理员可执行“驳回举报”。

驳回时需填写：

- 驳回原因/备注（必填）

处理完成后：

- 举报状态变为“已驳回”
- 记录处理人、处理时间、处理备注
- 不对内容做处置
- 通知举报人“举报未被采纳”

### 5.7 站内通知

举报处理完成后，系统给举报人发送一条通知。

通知文案示例：

- 举报已处理：你举报的照片经核查已处理，感谢反馈
- 举报已驳回：你举报的评论经核查暂不处理，原因：证据不足

说明：

- 一期只通知举报人
- 被举报内容作者暂不通知，二期再补

## 6. 用户流程

### 6.1 用户侧流程

```
用户浏览照片/评论
    │
    ▼
点击“举报”
    │
    ▼
选择举报原因 + 补充说明
    │
    ▼
提交举报
    │
    ▼
系统创建举报记录，状态=待处理
    │
    ▼
用户可在“我的举报”中查看状态
```

### 6.2 平台侧流程

```
管理员进入举报列表
    │
    ▼
查看待处理举报
    │
    ▼
进入举报详情
    │
    ├── 采纳举报 → 执行内容处置 → 记录处理结果 → 通知举报人
    │
    └── 驳回举报 → 记录驳回原因 → 通知举报人
```

### 6.3 闭环结果

```
用户提交举报
    │
    ▼
平台审核
    │
    ▼
内容被处理或举报被驳回
    │
    ▼
用户收到通知，并可在“我的举报”中查看最终结果
```

## 7. 状态机设计

举报状态 `status`：

- `0`：待处理
- `1`：已处理
- `2`：已驳回

状态流转规则：

- 待处理 → 已处理
- 待处理 → 已驳回
- 已处理 / 已驳回 不允许再次变更

说明：一期不支持“撤回处理”“重新打开举报”，避免状态过于复杂。

## 8. 数据模型

### 8.1 举报表 `t_report`

在现有基础上补齐处理字段。

| 字段 | 类型 | 说明 |
|------|------|------|
| id | BIGINT | 主键 |
| user_id | BIGINT | 举报人 ID |
| target_type | VARCHAR(20) | 举报对象类型：photo/comment |
| target_id | BIGINT | 举报对象 ID |
| reason | VARCHAR(50) | 举报原因 |
| description | VARCHAR(500) | 举报补充说明 |
| status | TINYINT | 0待处理 1已处理 2已驳回 |
| handle_result | VARCHAR(500) | 处理结果/驳回原因 |
| handled_by | BIGINT | 处理人 ID |
| handled_time | DATETIME | 处理时间 |
| create_time | DATETIME | 创建时间 |
| update_time | DATETIME | 更新时间 |

索引建议：

- `(status, create_time)`：管理端按状态分页
- `(user_id, create_time)`：用户查看我的举报
- `(target_type, target_id, status)`：重复举报校验与对象聚合

### 8.2 通知表复用

复用现有 `t_notification` 表，新增一种通知类型：

- `report_result`

## 9. 权限设计

一期采用“管理员白名单”方案，不引入完整 RBAC。

- 配置项：`admin.user-ids`
- 若当前登录用户的 `userId` 在白名单中，则可访问管理员举报接口
- 非管理员访问返回 403

原因：

- 改动小
- 落地快
- 足以支撑一期运营闭环

## 10. 接口设计

### 10.1 用户提交举报

```
POST /api/report/submit
```

请求参数：

```json
{
  "targetType": "photo",
  "targetId": 123,
  "reason": "色情低俗",
  "description": "图片包含明显不当内容"
}
```

返回：

```json
{
  "code": 200,
  "message": "success",
  "data": {
    "reportId": 9001,
    "status": 0
  }
}
```

### 10.2 获取我的举报列表

```
GET /api/report/my
```

参数：

- `page`
- `size`

返回：

```json
{
  "code": 200,
  "data": {
    "list": [
      {
        "id": 9001,
        "targetType": "photo",
        "targetId": 123,
        "reason": "色情低俗",
        "status": 1,
        "handleResult": "已下架违规照片",
        "createTime": "2026-03-10T10:00:00",
        "handledTime": "2026-03-10T11:00:00"
      }
    ],
    "total": 1,
    "hasMore": false
  }
}
```

### 10.3 获取管理员举报列表

```
GET /api/admin/report/list
```

参数：

- `status`：可选
- `targetType`：可选
- `page`
- `size`

### 10.4 获取管理员举报详情

```
GET /api/admin/report/detail/{id}
```

返回内容需包含举报信息、举报人信息、被举报对象信息、被举报对象作者信息。

### 10.5 采纳举报

```
POST /api/admin/report/resolve
```

请求参数：

```json
{
  "reportId": 9001,
  "action": "REMOVE_CONTENT",
  "handleResult": "经核查，内容违规，已下架"
}
```

规则：

- 仅允许处理待处理举报
- `handleResult` 必填
- 根据 `targetType` 自动执行内容处置

### 10.6 驳回举报

```
POST /api/admin/report/reject
```

请求参数：

```json
{
  "reportId": 9001,
  "handleResult": "经核查，证据不足，暂不处理"
}
```

规则：

- 仅允许处理待处理举报
- `handleResult` 必填

## 11. 页面与入口

### 11.1 用户侧

- 照片详情页举报入口：保留并升级
- 评论区举报入口：新增
- 个人中心增加“我的举报”入口
- 新增“我的举报”列表页

### 11.2 管理员侧

一期建议提供一个极简管理端页面，至少包含：

- 举报列表页
- 举报详情页

如果一期来不及做 Web 管理页，也必须提供完整管理 API，并保证可以通过接口工具完成审核处理。

说明：从产品角度，一期目标是闭环，不强制要求复杂后台 UI，但必须具备真实可操作的审核入口。

## 12. 交互细节

| 场景 | 行为 |
|------|------|
| 用户点击举报 | 弹出原因选择面板 |
| 用户提交成功 | Toast：“举报已提交，我们会尽快处理” |
| 重复未处理举报 | Toast：“你已举报过该内容，请等待处理” |
| 我的举报为空 | 显示“你还没有提交过举报” |
| 管理员进入列表 | 默认展示待处理举报 |
| 管理员采纳举报 | 二次确认后执行内容处置 |
| 管理员驳回举报 | 必须填写驳回原因 |
| 举报处理完成 | 举报人收到站内通知 |

## 13. 不做的事情（本期）

- 封禁用户
- 限流/禁言
- 举报申诉
- 批量审核
- 风险打分
- 自动审核
- 被举报人通知
- 运营数据看板
- 完整 RBAC 权限系统

## 14. 技术要点

- 举报对象校验：
  - `photo` 必须存在且未删除
  - `comment` 必须存在且未删除
- 管理员处置内容时优先使用逻辑删除
- 举报处理必须加事务，避免“状态已处理但内容未下架”
- 管理员接口统一放在 `/api/admin/**`
- 管理员鉴权可通过拦截器或权限组件统一封装
- 举报处理完成后，复用现有通知系统创建 `report_result` 通知
